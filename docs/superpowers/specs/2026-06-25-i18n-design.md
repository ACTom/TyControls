# ty-controls Internationalization (i18n) — Design

**Date:** 2026-06-25
**Status:** Approved (brainstorming) → ready for implementation plan

## Goal

Make the **translatable surface** of ty-controls internationalized using the standard Lazarus/FPC
mechanism (`resourcestring` + GNU-gettext `.po`), shipping **English (base) + Simplified Chinese
(zh_CN)**. Cover three surfaces: design-time IDE strings, theme diagnostics, and the main demo.

## Background — the key finding

A full audit (every `.pas` in `source/`, `designtime/`, `examples/`) found **221 user-facing string
literals, but the runtime control units contain ZERO of them.** Every control (Button, Edit, Memo,
Form, Menu, …) is custom-drawn and gets **all** of its visible text — captions, items, hints — from the
host application via published properties. The controls embed no prose. So "internationalizing
ty-controls" is **not** about the controls; the real translatable surface is:

| Surface | Count | Audience | Package/App |
|---|---|---|---|
| Design-time IDE strings | 18 | IDE user (design time) | `tycontrols_dt` (design-time pkg) |
| Theme diagnostics | ~48 | theme author / developer | `tycontrols` (runtime pkg) |
| Demo UI | ~6 code + .lfm captions | end user | `examples/demo` (app) |
| Runtime controls | **0** | — | — (nothing to do) |

(155 of the 221 were in throwaway per-control example apps — **out of scope**, see below.)

## Decisions (locked during brainstorming)

1. **Scope:** design-time IDE strings + theme diagnostics + the main demo. Runtime controls: nothing.
2. **Languages:** English is the base (`msgid`, the source literal); zh_CN is the translation (`.po`).
3. **Demo:** `examples/demo` **only** — the per-control mini-examples (`examples/button`, etc.) are
   throwaway test apps and stay as-is.
4. **Demo language switching:** a runtime **language switcher** (a dropdown beside the existing theme
   switcher) using `LCLTranslator.SetDefaultLang`; default follows the OS locale on startup.
5. **Diagnostics depth:** make **all** ~48 diagnostics `resourcestring`s (library becomes 100%
   translatable), but in `zh_CN.po` only translate the user-comprehensible ones (ThemeLint warnings +
   common theme-load errors). Deep CSS-syntax parser errors (e.g. "Unterminated :root block") keep an
   English `msgid` and may stay untranslated — gettext falls back to the `msgid`, and English is fine
   for that developer-facing detail.
6. **Architecture:** **central `*StrConsts` units** per package (the LCL `lclstrconsts.pas` model),
   not per-unit `resourcestring` blocks — fewer `.pot` files, all translatable text discoverable in one
   place, easier `.po` maintenance.
7. **Encoding:** UTF-8 end to end. Never disable `SystemCharSetIsUTF8`; never define
   `NoUTF8Translations`. `.po` headers declare `charset=UTF-8`.

## Mechanism (verified against the real Lazarus source at `C:\lazarus`)

- **Source:** user-facing literals become `resourcestring` constants. Composed messages
  (`'undefined variable -- ' + name`) are refactored to `Format(rsXxx, [name])` so Chinese can reorder.
- **`.po`/`.pot` generation:** enable the package/project **i18n option** (`EnableI18N=true`,
  `POOutputDirectory=languages`). After a build, FPC emits per-unit `<unit>.rsj`; the IDE merges them
  into `languages/<unit>.pot` (+ propagates to `<unit>.<lang>.po`) via `Translations.UpdatePoFile`.
  Naming is strict: `<unitname>.<langid>.po` (e.g. `tycontrols.strconsts.zh_CN.po`).
- **Runtime loading (demo):** `LCLTranslator.SetDefaultLang('zh_CN' | 'en')` detects/loads the `.po`,
  patches live `resourcestring`s (`Translations.TranslateResourceStrings`), installs the
  `TPOTranslator` LRS hook so every form's `TTranslateString` properties (Caption/Hint) translate as
  they stream, and `UpdateTranslation` re-walks already-open forms — so a runtime switch retranslates
  the whole UI. Startup default = OS locale.
- **Design-time loading (IDE):** the IDE reuses the **package's own `.po`** to translate design-time
  strings. `TPkgManager.DoTranslatePackage` runs at package registration (before `Register`), reading
  `EnvironmentOptions.LanguageID` and calling `TranslateUnitResourceStrings` per matching
  `<unit>.<lang>.po`. **Requires the package to set `POOutputDirectory`** and ship its `zh_CN.po`. This
  is exactly what makes the new File>New labels follow the IDE language.

## Components

### Foundation — shared rule
All `.po` UTF-8; `msgid` = English/ASCII; only `msgstr` carries CJK. Identifiers are lowercased
`<unit>.<rsname>` lookup keys. Format strings tagged `#, object-pascal-format`; a `%`-arg mismatch in a
translation is silently skipped (English shown) rather than crashing.

### Part ① — Runtime package: theme diagnostics (~48 strings)
- **New unit `source/tyControls.StrConsts.pas`** — a `resourcestring` block for all diagnostics:
  ThemeLint warnings (`tyControls.ThemeLint.pas`), CSS parser errors (`tyControls.Css.Parser.pas`),
  CSS value-eval errors (`tyControls.Css.Values.pas`), StyleModel declaration/`@import` errors
  (`tyControls.StyleModel.pas`).
- **Refactor** those four units to reference the consts; convert concatenations to `Format(rs…, [...])`.
- **`tycontrols.lpk`:** add the new unit; set i18n `EnableI18N=true`, `POOutputDirectory=languages`
  (`EnableI18NForLFM` not needed — no shipped runtime `.lfm`).
- **Artifacts:** `languages/tycontrols.strconsts.pot` (generated) + hand-authored
  `languages/tycontrols.strconsts.zh_CN.po` (translate the comprehensible subset).

### Part ② — Design-time package: IDE strings (18 strings)
- **`designtime/tyControls.Design.pas`:** add a `resourcestring` block for File>New item labels +
  descriptions (`GetLocalizedName`/`GetLocalizedDescription` of the 3 descriptors), the About dialog
  (`ShowTyAboutDialog` — `'About TyControls'`, the tagline `'Themed LCL control library'` (was hardcoded
  Chinese), `'Version '`, `'OK'`), and the component-editor verbs (`'Add Page'`, `'Delete Page'`,
  `'Show Next Page'`, `'Show Previous Page'`). Reference the consts at the use sites.
- **`tycontrols_dt.lpk`:** set i18n `EnableI18N=true`, `POOutputDirectory=languages`.
- **Artifacts:** `languages/tycontrols.design.pot` + `tycontrols.design.zh_CN.po`.
- Result: File>New labels/descriptions, the About dialog, and the page-editor verbs all follow the IDE
  UI language.

### Part ③ — Demo: `examples/demo` (English base + runtime switcher)
- **Code strings → `resourcestring`** in the demo units (`mainform.pas`, `chromeform.pas`): the
  theme-combo "Custom…" entry, the open-dialog filter label, the greeting captions, the context-menu
  caption, the chrome-window title. English `msgid`s.
- **`.lfm` captions → English** (menu items, button/label captions currently Chinese). Enable the
  project's `EnableI18NForLFM` so `.lrj` harvests them into the `.po` (no per-caption code change; the
  `TPOTranslator` retranslates them at form-stream + on switch).
- **Language switcher:** add a `LangCombo` (or two buttons) beside the existing theme/appearance
  switcher in `mainform.pas`; `OnChange` calls `LCLTranslator.SetDefaultLang('en' | 'zh_CN')`. On
  startup, call `SetDefaultLang('')` (autodetect OS) — add `LCLTranslator` to the demo `uses`.
- **`demo.lpi`:** enable project i18n (`EnableI18N`, `EnableI18NForLFM`, `POOutputDirectory=languages`).
- **Artifacts:** `examples/demo/languages/*.pot` + `*.zh_CN.po` (code resourcestrings + form captions).
- Optional polish: also load `lclstrconsts.zh_CN.po` so the LCL's own dialog buttons show Chinese
  (`SetDefaultLang` calls `TranslateLCLResourceStrings` automatically if the file is locatable).

## Testing

- **Headless unit test** (the runtime part is testable; design-time/demo are not): load
  `tycontrols.strconsts.zh_CN.po`, call `Translations.TranslateUnitResourceStrings('tyControls.StrConsts', po)`,
  assert a known const now reads its Chinese translation, then restore. Guards the .po↔resourcestring
  wiring + the UTF-8 round-trip. Add to the existing `tests/` suite (keep 951/0/… green).
- **Build verification:** both packages + the demo compile; the i18n option produces the `.pot` files.
- **Manual (real machine, like prior IDE/visual work):** IDE in Chinese shows translated File>New
  labels / About dialog / page-editor verbs; the demo language switcher flips the whole UI live.

## Phasing (independent, land in order)
1. **Foundation + Part ① (diagnostics):** new `StrConsts` unit, refactor the 4 units, `.lpk` i18n,
   `.pot`/`.po`, headless test. Self-contained + testable.
2. **Part ② (design-time):** Design.pas resourcestrings, `_dt.lpk` i18n, `.po`.
3. **Part ③ (demo):** resourcestrings + `.lfm` English + switcher + project i18n + `.po`.

## Risks / not testable here
- IDE design-time translation (`DoTranslatePackage`) and the demo live-switch are **only verifiable in a
  real Lazarus IDE / GUI** — same constraint as prior design-time + window-effects work. The headless
  test + build are the automated guard; the IDE/visual check is manual.
- `.po` filename convention is strict (`<unit>.<langid>.po`, lowercased keys) — get it wrong and nothing
  loads. Model filenames on LCL's `lcl/languages/` + fpcunit's `components/fpcunit/languages/`.
- Refactoring exception literals to `resourcestring`/`Format` must preserve `%`-arg order/count
  (mismatch → translation silently skipped). Keep `#, object-pascal-format` discipline in the `.po`.
